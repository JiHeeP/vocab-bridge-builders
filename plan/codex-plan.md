# Codex 실행 계획

## 전제
- 목표는 `Supabase` 의존성을 제거하고 `Railway PostgreSQL + Express`로 단일 서비스 구조를 만드는 것이다.
- 현재 저장 계층은 `students`, `learning_records`, `intervention_logs`, `word_images` 4개 테이블에 집중되어 있다.
- 현재 프론트엔드는 조회/집계를 상당 부분 클라이언트에서 직접 수행한다. 이 점을 이용하면 1차 마이그레이션에서 서버 집계 API를 최소화할 수 있다.

## Claude 계획에서 유지할 부분
- Express 서버를 추가하고 `dist/` 정적 파일까지 같은 프로세스에서 서빙하는 방향
- `pg` 기반 Railway PostgreSQL 연결
- 프론트엔드의 Supabase 호출을 `fetch('/api/...')` 기반으로 치환
- Supabase 관련 파일, 의존성, 배포 설정 제거

## 제가 조정하는 부분
- 14개 엔드포인트를 한 번에 만들지 않는다.
- 1차 목표는 "현재 UI가 실제로 호출하는 기능"만 안정적으로 치환하는 것이다.
- 그룹 리포트 집계는 당분간 클라이언트 계산을 유지한다. 서버에서 복잡한 집계 쿼리를 먼저 만들 필요가 없다.
- 이미지 가져오기 기능은 외부 API 의존성이 있으므로 마지막 단계로 분리한다.

## 실제 구현 순서

### 1. 베이스라인 고정
- Supabase 사용 위치를 기준으로 교체 대상 파일을 확정한다.
- 기존 스키마를 바탕으로 PostgreSQL 초기화 SQL을 서버에 내장하거나 별도 SQL 파일로 분리한다.
- `package.json`, `railway.json`, `.env.example` 수준의 최종 실행 구조를 먼저 정의한다.

### 2. 서버 골격 작성
- `server/index.ts`: Express 앱, JSON 파서, CORS, 헬스체크, `dist/` 정적 서빙, SPA fallback
- `server/db.ts`: `pg.Pool` 초기화, 연결 테스트, 시작 시 스키마 보장
- `server/types.ts` 또는 유사 파일: 프론트와 서버가 공유할 최소 타입 정의
- 이 단계가 끝나면 Railway에서 "서버는 뜨고 정적 파일은 서빙되는 상태"까지 확보한다.

### 3. 핵심 CRUD API 먼저 이관
- 학생 API: 목록, 추가, 수정, 삭제, 이름 기반 get-or-create
- 학습 기록 API: 저장, 학생별 조회, 전체 조회
- 중재 기록 API: 조회, 생성
- 이 단계에서는 현재 UI 흐름을 그대로 살리는 데 필요한 응답 형태만 맞춘다.

### 4. 프론트엔드 데이터 접근 계층 통합
- `src/lib/api.ts`를 만들고 모든 네트워크 호출을 여기로 통일한다.
- `src/lib/scoreService.ts`부터 먼저 교체한다. 여기서 학생 조회/생성, 학습 기록 저장/조회, 중재 기록 저장/조회가 모두 엮여 있다.
- 이후 `DashboardPage.tsx`, `StudentManagement.tsx`를 교체한다.
- 이 단계 목표는 학습 저장과 대시보드 조회가 다시 동작하는 것이다.

### 5. 단어 이미지 기능 이관
- `word_images` 조회 API를 먼저 붙여 기존 이미지 표시 기능을 복구한다.
- `VocabManagement.tsx`는 이미지 존재 여부 조회부터 이관한다.
- 외부 이미지 수집은 마지막에 붙인다. 이유는 다음과 같다:
- 현재 Supabase Edge Function은 이미지 검색 + 번역 + 캐시 저장을 한 번에 담당한다.
- 이 부분은 환경변수, 외부 네트워크, 실패 재시도 정책이 필요하므로 CRUD보다 리스크가 크다.
- 가능하면 `POST /api/word-images/fetch`는 별도 서비스 함수로 분리해 서버 라우트는 얇게 유지한다.

### 6. Supabase 제거 및 배포 구성 정리
- `@supabase/supabase-js`, `serve` 제거
- `express`, `pg`, `cors`, `tsx` 추가
- `src/integrations/supabase/*` 삭제
- `supabase/` 폴더는 실제로 더 이상 참조되지 않는지 확인 후 제거
- `railway.json`은 `tsx server/index.ts` 기동으로 전환

### 7. 검증
- `npm run build`
- `npx tsx server/index.ts`
- 학생 추가/수정/삭제
- 학습 기록 저장 후 대시보드 반영
- 중재 기록 저장/조회
- 이미지 조회 및 이미지 수집 기능 확인

## 1차 마이그레이션 완료 기준
- 프론트가 더 이상 Supabase SDK를 import하지 않는다.
- Railway 한 서비스에서 프론트 정적 파일과 API를 모두 제공한다.
- 학생 관리, 학습 저장/조회, 대시보드 조회가 동작한다.
- 이미지 조회는 동작하고, 이미지 수집은 최소한 수동 실행 가능하다.

## 주요 리스크
- 현재 UI 문자열 일부가 이미 깨져 보인다. 마이그레이션과 별개로 인코딩 점검이 필요할 수 있다.
- 이미지 수집 기능은 외부 API 키와 호출 제한 정책이 없으면 배포 후 불안정할 수 있다.
- 서버 시작 시 자동 테이블 생성 방식을 택하면 구현은 빠르지만, 이후 마이그레이션 관리 체계는 별도로 정리해야 한다.

## 제가 다음 작업에서 바로 할 일
1. `server/` 골격과 DB 연결부터 만든다.
2. `scoreService`가 기대하는 학생/학습기록/중재기록 API를 먼저 맞춘다.
3. 프론트를 `api.ts` 기반으로 옮긴 뒤 빌드와 로컬 실행까지 확인한다.
4. 마지막으로 이미지 기능과 배포 설정을 정리한다.
